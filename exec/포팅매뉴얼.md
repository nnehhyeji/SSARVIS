# SSARVIS 포팅 매뉴얼

## 1. 문서 목적

- 이 문서는 GitLab 소스 클론 이후, 개발/운영 환경에서 SSARVIS를 빌드하고 배포하기 위한 기준 문서다.
- 저장소 구조는 `frontend`, `backend`, `ai`, `infra`, `compose-front` 로 나뉜다.
- 실제 운영 배포는 `infra/deploy` 기준의 Docker Compose 스택이며, 로컬 통합 실행은 `compose-front/docker-compose.yml` 조합이 가장 빠르다.

## 2. 프로젝트 구성 요약

| 영역 | 주요 기술 | 역할 |
| --- | --- | --- |
| frontend | React 19, TypeScript 5.9, Vite 7, Tailwind CSS 4 | 웹 UI |
| backend | Spring Boot 3.5.11, Java 21, Gradle 8.14.4 | 인증, API, WebSocket, DB 연동 |
| ai | FastAPI, Python 3.11, Uvicorn, uv | OpenAI/DashScope/Qdrant 기반 AI/음성 처리 |
| infra | Nginx, Docker Compose, Elastic Stack(APM/Kibana/Logstash/Elasticsearch) | 정적 웹 배포, 리버스 프록시, 모니터링, 통합 배포 |
| data services | MySQL 8.0.45, Redis 8.4.2, MongoDB 8.2, Qdrant latest | 영속 저장소, 캐시, 문서 저장, 벡터 DB |

## 3. 권장 개발 환경

### 3.1 명시적으로 확인되는 버전

- Docker / Docker Compose plugin 필요
- Java 21
- Gradle 8.14.4 이상
- Python 3.11
- Node.js 24 계열
- npm 사용

### 3.2 IDE

- 저장소에는 `.idea`, `.vscode`, `.iml` 등 IDE 고정 버전 파일이 커밋되어 있지 않다.
- 따라서 아래는 명시 버전이 아니라 권장안이다.
  - Backend: IntelliJ IDEA 2024.3 이상
  - Frontend: WebStorm 또는 VS Code 최신 안정판
  - AI: PyCharm 또는 VS Code 최신 안정판

## 5. 소스 클론 이후 로컬 실행 절차

### 5.1 저장소 클론

```bash
git clone <gitlab-repository-url>
cd ssarvis
```

### 5.2 AI/Backend 인프라 포함 로컬 통합 실행

`compose-front/docker-compose.yml` 기준으로 실행한다.

```bash
cd compose-front
docker compose up -d --build
```

구성 서비스

- MySQL
- Redis
- MongoDB
- Backend
- AI
- Qdrant

기본 노출 포트

- MySQL `3306`
- Redis `6379`
- MongoDB `27017`
- Backend `8080`
- AI `18000 -> 8000`
- Qdrant `16333 -> 6333`

### 5.3 Frontend 로컬 실행

프런트는 `compose-front`에 포함되어 있지 않으므로 별도 실행한다.

```bash
cd frontend
npm ci
npm run dev
```

기본 필요 환경 변수

```env
VITE_API_BASE_URL=http://localhost:8080
VITE_KAKAO_REST_API_KEY=<kakao_javascript_or_rest_key>
VITE_KAKAO_OAUTH_REDIRECT_URL=http://localhost:5173/auth/kakao/callback
```

주의

- 프런트 코드에서 Kakao 인증 시작 URL을 직접 생성하므로 `VITE_KAKAO_REST_API_KEY`, `VITE_KAKAO_OAUTH_REDIRECT_URL` 이 누락되면 소셜 로그인 버튼이 동작하지 않는다.
- 마이크 기능을 사용하므로 브라우저 `microphone` 권한 허용이 필요하다.

## 6. 운영/배포 절차

### 6.1 운영 배포 방식 요약

운영용 스택은 `infra/deploy/scripts/deploy.sh` 를 기준으로 동작한다.

배포 순서

1. Docker 네트워크 생성
2. Docker Hub 로그인 선택 수행
3. 애플리케이션 이미지 pull
4. DB 스택 기동
5. 모니터링 스택 기동
6. 앱 스택 기동
7. `notification_types` 초기 데이터 seed

### 6.2 운영 배포 전 준비 파일

`infra/deploy/env` 아래 파일들이 필요하다. 현재 저장소에는 실제 파일이 커밋되어 있지 않으므로 배포 서버에서 별도 생성해야 한다.

- `app.env`
- `app.secret.env`
- `db.env`
- `db.secret.env`
- `ai.env`
- `ai.secret.env`
- `monitoring.env`
- `monitoring.secret.env`

실제 예시는 `compose-front/.env`, `ai/.env.example`, `backend/src/main/resources/application.yaml` 를 기준으로 재구성할 수 있다.

### 6.3 이미지 기반 운영 배포

운영 스크립트는 소스 빌드보다 이미지 pull 방식을 전제로 한다.

필요 예시 변수

```env
BACKEND_IMAGE=<registry/backend-image>
BACKEND_TAG=<tag>
AI_IMAGE=<registry/ai-image>
AI_TAG=<tag>
WEB_IMAGE=<registry/web-image>
WEB_TAG=<tag>
DOCKERHUB_USERNAME=<dockerhub-user>
DOCKERHUB_TOKEN=<dockerhub-token>
```

실행

```bash
cd infra/deploy/scripts
./deploy.sh
```

### 6.4 소스 기준으로 직접 이미지 빌드해야 하는 경우

운영 서버에서 이미지가 없으면 아래처럼 먼저 빌드한 후 태그를 `app.env`, `ai.env` 등에 맞춰 사용한다.

```bash
docker build -f backend/Dockerfile -t <BACKEND_IMAGE>:<BACKEND_TAG> backend
docker build -f ai/Dockerfile -t <AI_IMAGE>:<AI_TAG> ai
docker build -f infra/nginx/Dockerfile -t <WEB_IMAGE>:<WEB_TAG> .
```

## 9. 배포 시 특이사항

- 운영 배포 스크립트는 소스 빌드가 아니라 registry 이미지 pull 중심이다.
- `infra/deploy/scripts/init-network.sh` 가 외부 네트워크를 생성하므로 Docker 권한이 필요하다.
- Nginx가 `80`, `443/tcp`, `443/udp` 를 사용하므로 서버 보안그룹/방화벽을 열어야 한다.
- SSL 인증서 파일이 `infra/deploy/nginx-certs` 또는 compose에서 바라보는 경로에 준비되어 있어야 한다.
- `infra/nginx/.htpasswd` 가 없으면 Kibana basic auth 설정이 깨질 수 있다.
- Backend는 기동 후 health check `http://localhost:8080/actuator/health` 를 통과해야 Nginx upstream 대상으로 간주할 수 있다.
- AI는 시작 시 Qdrant 컬렉션 초기화 및 Local TTS preload 를 수행한다.
- Backend의 JPA `ddl-auto: update` 이므로 운영에서 스키마 변경 시 자동 반영 부작용을 검토해야 한다.
- 배포 완료 후 `notification_types` 테이블에 `FOLLOW_REQUEST`, `FOLLOW_ACCEPT` 가 seed 된다.
- Frontend 운영 이미지는 `infra/nginx/Dockerfile` 에서 빌드되므로, 프런트 산출물과 Nginx 설정이 한 이미지에 결합된다.
- Kakao OAuth redirect URI는 프런트와 백엔드 모두 정확히 맞아야 한다.
- Gmail SMTP 사용 시 일반 비밀번호가 아니라 앱 비밀번호가 필요하다.
- `compose-front/.env` 에는 실제 민감값이 포함될 수 있으므로 운영 문서에서는 값 자체가 아니라 변수명과 관리 위치만 공유해야 한다.

## 10. 확인이 필요한 항목

- 실제 운영 서버의 `infra/deploy/env/*.env` 실파일은 저장소에 없으므로, 최종 배포값은 서버 내 파일과 비교 확인이 필요하다.
- IDE 버전은 저장소에 명시되어 있지 않으므로 팀 운영 표준이 있다면 해당 버전으로 별도 보완해야 한다.
- AI 모델명은 예시 파일과 코드 기본값이 다를 수 있다.
  - 예시: `gpt-4o-mini`
  - 코드 기본값: `gpt-5.4-mini`
  실제 배포 전 반드시 하나로 통일해야 한다.
